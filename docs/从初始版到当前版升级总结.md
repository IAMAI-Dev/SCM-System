# 供应链管理系统：从初始版到当前版的功能与技术升级总结

> 对比基线：旧版 `scm_gui.py` 与当前以 `main.py` 为入口的 SCM-System  
> 总结日期：2026 年 6 月 18 日

## 一、总结结论

当前系统相比最初的 `scm_gui.py`，已经不再是一个仅用于展示数据库增删改查的单文件程序，而是发展成了一个具备登录认证、角色权限、业务分层、事务处理、操作审计、数据分析、Excel 导出、数据库迁移和性能优化能力的桌面供应链管理系统。

这次演进可以概括为五个核心变化：

1. **从单文件原型升级为分层工程**：由一个 327 行左右的 Tkinter 文件，发展为约 44 个 Python 文件、约 5053 行 Python 代码和 7 个 SQL 迁移脚本组成的工程。
2. **从四个基础页面扩展为完整业务平台**：在订单、库存、供应商和客户功能之外，增加了登录、仪表盘、采购管理、经营分析、性能监控、用户权限和审计日志等模块。
3. **从直接操作数据库升级为受控业务流程**：写操作统一经过 Service 层权限校验、参数验证、事务处理和审计记录。
4. **从静态表格升级为分析型界面**：增加 KPI、趋势图、分布图、排名、低库存预警、ABC 分类和客户价值分析。
5. **从“能运行”升级为“可配置、可迁移、可维护、可验证”**：增加配置管理、连接池、版本化数据库迁移、索引、冒烟检查和运维文档。

因此，新版的本质不是单纯更换 GUI 框架，而是完成了从“课程演示式 CRUD 工具”到“具有企业应用基本工程特征的供应链桌面系统”的升级。

## 二、对比范围与依据

本总结以以下两部分为主要对比对象：

- **初始版**：根目录中的 `scm_gui.py`，采用 Tkinter，全部界面、业务逻辑和 SQL 写在同一个文件中。
- **当前版**：以 `main.py` 为入口，代码分布在 `core/`、`dao/`、`service/`、`ui/`、`sql/`、`utils/` 和 `tools/` 中。

结论依据不仅包括 README 和 Changelog，也核验了当前页面、Service、DAO、SQL 迁移和 Git 提交记录。文档中对“智能补货”“性能监控”等名称按实际代码能力描述，避免把规则功能夸大为人工智能，或把普通索引清单误写成精确的未使用索引分析。

## 三、系统规模变化

| 指标 | 初始版 | 当前版 | 变化说明 |
| --- | ---: | ---: | --- |
| Python 主体文件 | 1 个 | 约 44 个 | 从单文件拆分为多个职责清晰的模块 |
| Python 代码规模 | 约 327 行 | 约 5053 行 | 增加完整业务、架构和基础设施能力 |
| SQL 脚本 | 0 个独立脚本 | 7 个迁移脚本，约 388 行 | 数据库对象实现版本化管理 |
| 主业务入口 | `scm_gui.py` | `main.py` | 旧文件只保留为历史参考 |
| GUI 技术 | Tkinter | PySide6 / Qt6 | 界面、模型、线程和交互能力全面升级 |
| 主要页面 | 4 个 | 10 个导航模块及欢迎页 | 从基础操作扩展到管理与分析平台 |

当前 10 个导航模块包括：总览仪表盘、订单管理、库存调度、客户管理、供应商分析、用户权限、审计日志、性能监控、经营分析和采购管理。不同角色看到的入口不同。

## 四、初始版具备的功能

初始版 `scm_gui.py` 的主要能力如下：

### 1. 订单管理

- 展示前 100 条订单。
- 按订单号搜索。
- 查看订单对应的 Lineitem 明细。
- 手工更新订单状态。

### 2. 库存查询

- 联结 Part、PartSupp 和 Supplier 表。
- 展示零件、品牌、库存量、供应商和成本价。
- 仅支持查询，没有库存预警、分类、补货和采购单流程。

### 3. 供应商分析

- 按供应商统计供应零件数量和平均成本。
- 结果仅以普通表格显示，没有 KPI、趋势、分布图、评分或排名。

### 4. 客户管理

- 展示和搜索客户。
- 新增客户。
- 删除客户。
- 尚未实现完整的客户编辑、导出和细粒度权限控制。

### 5. 初始版的技术局限

- UI、业务规则和 SQL 全部耦合在一个类中。
- 启动时直接创建单个数据库连接和游标。
- 没有登录、用户、角色、部门或权限模型。
- 没有事务上下文，多个关联写操作无法统一提交或回滚。
- 没有应用级审计日志和订单状态审计。
- 没有配置文件、连接池和数据库版本迁移。
- 数据查询发生在 GUI 主线程，数据量增大时容易造成界面卡顿。
- 缺少统一异常封装、自动检查和部署运维说明。

## 五、新增和增强的业务功能

### 1. 登录认证与欢迎页

新版增加了独立登录窗口和登录后的轻量欢迎页：

- 按用户名和密码认证启用状态的用户。
- 密码不再直接按明文比较，而是在 MySQL 中通过 SHA-256 摘要匹配。
- 登录成功后创建不可变的 `UserSession`，保存用户、部门、角色和动作权限。
- 登录行为自动写入审计日志。
- 登录后先展示欢迎页，不立即加载全部业务数据，从而缩短首屏等待时间。

### 2. 多角色、部门和动作权限体系

系统新增了三类角色和三个业务部门：

| 角色 | 主要权限 |
| --- | --- |
| admin | 可访问并操作全部模块，管理用户权限，查看性能监控 |
| manager | 可查看跨部门业务和经营分析，只修改所属部门业务，可查看日志 |
| staff | 只进入所属部门模块，按权限执行查看或新增，不能更新和删除 |

部门分为采购部、销售部和客户管理部。权限粒度包含查看、新增、更新和删除。

权限控制形成了两道防线：

- UI 根据会话决定是否显示页面、启用按钮。
- Service 层在真正执行写操作前再次校验权限。

这解决了仅靠“隐藏按钮”容易被绕过的问题。管理员权限在后端保持全开，员工的更新和删除权限也会被后端强制关闭。

### 3. 总览仪表盘

初始版没有统一总览。新版增加：

- 订单总数。
- 总营收。
- 客户总数。
- 低库存数量。
- 销售趋势数据。
- 低库存预警表。
- 低库存预警 Excel 导出。

KPI 由数据库存储过程和专用 DAO 提供，仪表盘查询通过后台线程执行，页面可以显示加载、成功和失败状态。

### 4. 订单管理增强

在原有订单列表、搜索、明细和状态更新基础上，新增：

- 支持按订单号或客户名称模糊查询。
- 展示订单优先级和更完整的明细字段。
- 支持创建包含一条明细的模拟订单。
- 对客户、零件、供应商、数量以及供应关系进行业务校验。
- 订单头和订单明细在同一事务中创建。
- 状态更新和模拟下单写入操作日志。
- 数据库触发器单独记录订单状态的旧值、新值和变更时间。
- 当前订单列表可导出为 Excel。
- 不同部门和角色拥有不同的查询、创建和更新能力。

### 5. 库存调度增强

初始版只有库存查询；新版增加：

- 库存状态分级：低库存、预警、正常。
- “只看低库存”筛选。
- 手工补货并即时更新库存数量。
- 对零件 ID、供应商 ID、补货数量和供应关系进行校验。
- 按库存价值累计占比进行 ABC 分类。
- 根据库存低于 100 的规则生成补货建议。
- 从库存记录直接生成补货采购单。
- 补货和采购单创建均记录操作日志。
- 业务更新与日志写入在同一事务中提交或回滚。

需要准确说明的是，当前“智能补货建议”属于阈值规则：目标库存为 100，建议数量为 `100 - 当前库存`；它不是机器学习预测或复杂需求预测模型。

### 6. 采购管理

系统新增 `scm_replenishment_orders` 补货采购单表和采购管理页面：

- 从库存页面生成采购单。
- 记录零件、供应商、数量、状态和创建时间。
- 按时间倒序查看采购单。
- 采购部人员按角色权限访问。

当前采购管理主要完成了“生成 + 列表查看”的基础闭环，尚未实现审批、状态流转、到货入库和采购单撤销等完整采购生命周期。

### 7. 客户管理增强

在查询、新增和删除基础上，新增：

- 支持按客户 ID 或名称搜索。
- 支持编辑客户。
- 展示订单数量、累计营收等客户汇总信息。
- 新增、编辑、删除均执行输入校验和权限校验。
- 客户写操作与审计日志使用同一事务。
- 客户列表可导出为 Excel。
- 首屏查询先限制客户范围再聚合订单，减少全表聚合等待。
- 表格采用整行选择、固定行高和状态提示，改善大量数据下的使用体验。

### 8. 供应商分析增强

旧版只展示供应件数和平均成本，新版增加了较完整的供应商分析驾驶舱：

- 活跃供应商、优质供应商、供应零件总数、月均采购额等 KPI。
- 供应商供应量对比柱状图。
- 供应商国家分布环形图。
- 采购额趋势折线图。
- 供应商评分、排名和趋势展示。
- 排名数据导出为 Excel。
- 图表悬停提示、中文字体适配和长标签缩略。
- 窗口较小时自动把 KPI 和图表改为纵向紧凑布局。
- 供应商分析查询在后台线程运行，图表分阶段渲染，避免主线程长时间阻塞。

### 9. 经营分析驾驶舱

新版为经理和管理员增加经营分析页面：

- 最近 12 个月月度营收趋势。
- Top 10 畅销零件。
- 客户余额与订单数量的价值分布。
- 加载状态、错误状态和刷新能力。
- 后台查询与主线程分阶段绘图。

相关查询对数据范围进行了控制，例如客户价值分析只抽取有限客户样本后再关联订单，防止首次进入页面时进行无边界全量聚合。

### 10. 用户权限管理

新增管理员专用用户管理页面：

- 查看用户、角色、部门和启用状态。
- 调整查看、新增、更新和删除权限。
- 保护管理员账户不会因错误操作失去全部管理权限。
- 强制员工不能获得更新和删除权限。
- 权限变化写入审计日志。

### 11. 审计日志

新版增加两类审计能力：

- `scm_logs`：记录登录、用户权限修改、订单创建、订单状态更新、库存补货、采购单创建和客户变更等关键行为。
- `scm_order_audit`：由触发器记录订单状态变更前后的值和时间。

经理和管理员可进入日志页面查看操作人、模块、动作、详情和发生时间。相比初始版，系统具备了基本的操作追踪和问题追责能力。

### 12. 数据库性能监控

新增管理员专用性能监控页面：

- 查看当前数据库各表的空间占用。
- 查看 MySQL 估算行数。
- 查看非主键索引清单。

该功能适合作为 DBA 调优的辅助入口。当前代码列出的是“非主键索引”，并没有结合 MySQL 使用统计准确判断索引是否从未使用，因此应避免把它表述为严格的“无效索引检测”。

### 13. Excel 导出

系统增加基于 pandas 和 openpyxl 的通用导出工具，可将 QTableView 当前数据保存为带时间戳的 `.xlsx` 文件。目前已用于：

- 订单报表。
- 客户报表。
- 低库存预警报表。
- 供应商排名报表。

导出依赖在需要时加载，减少页面首次创建的额外开销。

## 六、技术与架构升级

### 1. GUI：Tkinter 升级为 PySide6

PySide6 提供了比 Tkinter 更完整的企业桌面应用能力：

- `QMainWindow`、`QStackedWidget` 和独立业务页面。
- Model/View 表格模型，统一使用 `DictTableModel` 展示字典数据。
- QSS 统一主题和组件样式。
- 信号与槽机制。
- `QThread` 后台查询。
- 自定义对话框、进度条、滚动区和响应式布局。
- 与 Matplotlib QtAgg 画布集成。

### 2. 从单文件到分层架构

当前系统采用 `UI -> Service -> DAO -> MySQL` 的主要调用链：

```text
用户操作
   ↓
UI 页面：负责控件、交互、加载状态和数据展示
   ↓
Service：负责权限、参数校验、业务规则、事务编排和审计
   ↓
DAO：负责参数化 SQL、视图和存储过程调用
   ↓
MySQL：基础业务表、应用表、视图、过程、触发器和索引
```

辅助模块职责如下：

| 目录 | 主要职责 |
| --- | --- |
| `core/` | 配置、连接池、游标、事务、迁移和数据库异常 |
| `dao/` | 参数化 SQL 和数据库读写 |
| `service/` | 权限校验、业务规则、事务编排和审计 |
| `ui/` | 主窗口、页面、表格模型、图表和样式 |
| `sql/` | 有序数据库迁移 |
| `utils/` | Excel 导出等通用能力 |
| `tools/` | 静态冒烟检查 |

该架构使界面不再直接拼接 SQL，也让不同模块可以独立维护和测试。

### 3. 配置管理

初始版的连接参数直接写在 Python 文件附近；新版增加：

- `config.ini.example` 配置模板。
- 本地 `config.ini`，并由 Git 忽略真实配置。
- `SCM_DB_HOST`、`SCM_DB_PORT`、`SCM_DB_USER`、`SCM_DB_PASSWORD`、`SCM_DB_NAME` 等环境变量覆盖。
- 仓库名称、账套名称、主题、连接池名称和大小配置。
- 数据库名可按团队成员环境调整，顶部账套名称可跟随配置。

### 4. 数据库连接池与上下文管理

新版使用 mysql-connector-python 连接池，而不是长期复用一个全局连接和游标：

- `cursor()` 管理普通查询连接的获取和释放。
- `transaction()` 管理写操作的提交、异常回滚和资源释放。
- 统一封装数据库异常，向 UI 提供更可读的错误信息。
- 写操作可以在同一连接中同时完成业务变更和日志记录。

### 5. 参数化查询与输入校验

DAO 层普遍使用 `%s` 占位参数，不把外部输入直接拼进 SQL。Service 层进一步校验：

- ID 是否为正整数。
- 数量是否大于零。
- 状态值是否合法。
- 客户、零件、供应商及其关联关系是否存在。
- 当前用户是否能操作对应模块。

这同时提升了安全性、错误可控性和业务数据质量。

## 七、数据库工程能力新增

### 1. 应用辅助表

新增数据库对象包括：

- `scm_users`：用户、角色、部门和动作权限。
- `scm_logs`：应用操作日志。
- `scm_order_audit`：订单状态变更记录。
- `scm_replenishment_orders`：补货采购单。
- `scm_schema_migrations`：数据库迁移版本记录。

### 2. 数据库视图

新增四个主要业务视图：

- `v_scm_inventory_status`：统一库存、供应商和库存状态。
- `v_scm_daily_sales`：按日汇总订单和营收。
- `v_scm_supplier_performance`：供应商供应件数、成本和库存汇总。
- `v_scm_customer_summary`：客户订单数和累计营收汇总。

视图将重复统计逻辑下沉到数据库，减少不同 DAO 重复编写复杂联结。

### 3. 存储过程和触发器

- `sp_scm_dashboard_kpis` 集中返回仪表盘核心指标。
- 用户新增、用户修改触发器自动记录系统日志。
- 订单状态更新触发器自动写入状态审计表。

### 4. 版本化数据库迁移

当前 7 个脚本按顺序管理数据库变化：

1. 应用辅助表。
2. 业务视图。
3. 仪表盘存储过程。
4. 审计触发器。
5. 演示账号和兼容字段。
6. 补货采购单表。
7. 经营分析和趋势查询索引。

启动时通过 `scm_schema_migrations` 只执行尚未记录的脚本。已有数据库在对象完整时可建立迁移基线，新数据库按顺序初始化。相比每次启动重复建表或手工复制 SQL，这种机制更适合团队协作和版本升级。

### 5. 性能索引

当前增加了面向常用关联和分析查询的索引，例如：

- `Orders.Custkey`。
- `Lineitem.Shipdate`。
- `Lineitem.Receiptdate`。
- `Lineitem.Partkey`。
- `Lineitem.Suppkey`。
- `PartSupp.Suppkey`。

索引脚本先查询 `information_schema.statistics`，仅在索引不存在时创建，兼容已经手工建立索引的数据库。

## 八、界面、交互和性能优化

### 1. 统一工业控制台风格

新版使用深色侧边栏、浅色工作区、橙色主强调色和青色次强调色，形成统一 QSS 主题。界面增加：

- 品牌区、模块导航、顶部信息栏和用户身份区。
- 卡片、KPI、图表、状态提示和交替行色。
- 统一按钮、输入框、表格和滚动条样式。

### 2. 无边框窗口和原生缩放体验

主窗口实现：

- 自定义最小化、最大化和关闭按钮。
- 顶部栏拖动和双击最大化。
- Windows `WM_NCHITTEST` 原生边缘缩放。
- 子控件边缘鼠标追踪和对应缩放光标。
- 最小尺寸 `980×620`，默认尺寸 `1320×840`。

### 3. 页面懒加载

登录后只创建欢迎页和页面占位容器。业务页面在第一次点击时才创建并查询数据。这减少了启动时同时加载所有表格和图表的开销。

### 4. 后台查询和分阶段绘图

仪表盘、供应商分析和经营分析使用 `QThread` Worker 查询数据库：

- Worker 线程只负责数据查询。
- Qt 控件和 Matplotlib 绘制留在主线程。
- 通过信号传回结果或错误。
- 图表通过 `QTimer` 错峰渲染，避免一次绘制多个复杂图形。
- 刷新失败时可保留上一版有效数据，并给出固定状态提示。

### 5. 查询范围优化

当前版本对多个明显慢点进行了处理：

- 客户首屏先限制客户，再聚合对应订单。
- Top 零件先在 Lineitem 中聚合并限制前 10，再关联 Part。
- 月度营收限制最近 12 个月。
- 客户价值分布限制样本规模。
- 库存列表和日志列表设置首屏上限。
- 增加关联字段和日期字段索引。

## 九、安全性、可靠性和质量保障

### 1. 安全性提升

- 配置模板不保存真实密码，真实 `config.ini` 不进入 Git。
- 密码以 SHA-256 摘要形式存储和匹配，优于初始版无认证状态。
- 外部输入通过参数化 SQL 传入。
- Service 层执行真实权限校验。
- 关键操作可审计。
- 员工、经理和管理员的权限边界明确。

需要注意：当前 SHA-256 方案没有体现每用户随机盐和慢哈希。在真实生产环境中，更推荐 Argon2、bcrypt 或 PBKDF2，并增加登录失败次数限制、密码修改和会话超时。

### 2. 事务一致性

以下操作已经采用事务：

- 订单及明细创建。
- 库存补货。
- 补货采购单创建。
- 客户新增、更新和删除。
- 用户权限变更。

业务变更和对应日志在同一事务中提交，任何一步失败即可整体回滚。

### 3. 本地验证能力

新增 `tools/smoke_check.py`，用于检查：

- 关键目录和文件是否存在。
- UI 层是否意外直接使用数据库连接。
- 配置中是否出现明显的硬编码真实密码。
- 重要模块是否可以被静态扫描。

项目还提供编译检查、依赖检查、真实 MySQL 联调和 GUI 截图验证说明，并形成 README、项目规范、实施计划和运维文档。

## 十、项目演进过程

根据 Git 提交记录，项目大致经历了以下阶段：

1. **初始原型阶段**：完成 Tkinter 单文件订单、库存、供应商和客户功能。
2. **工程骨架阶段**：建立 PySide6 项目入口和 `core/dao/service/ui/sql` 目录。

6. **分析与权限强化阶段**：增加供应商可视化、多角色部门模型并收紧 Service 权限。

## 十一、新版带来的实际价值

## 十二、当前仍存在的边界与后续方向

虽然当前版已经完成大幅升级，但仍有以下可继续完善之处：

1. **补货建议仍是固定阈值规则**：可结合历史销量、交货周期、安全库存和季节性建立预测模型。
2. **采购流程尚未完整**：可增加审批、下单、在途、到货、入库、取消和供应商确认状态。
3. **订单主键生成仍偏应用逻辑**：高并发环境可改为自增键、序列表或更稳健的编号服务。
4. **部分列表仍以 LIMIT 控制首屏**：可增加服务端分页、排序和组合筛选。
5. **自动化测试仍较轻量**：可建立独立测试数据库，为 Service、DAO 和迁移增加可重复运行的集成测试。
6. **迁移机制可继续增强**：增加脚本校验和、失败恢复策略和迁移前自动备份提示。
7. **认证安全可继续加强**：使用带盐慢哈希、密码修改、失败锁定和会话超时。
8. **性能监控仍是基础视图**：可接入 `EXPLAIN`、慢查询日志、真实索引使用统计和查询耗时基线。
9. **顶部全局搜索目前主要是界面入口**：可进一步接通订单、客户、零件和供应商的跨模块搜索。
10. **采购单页面当前以查看为主**：可补充状态更新、筛选、导出和详情操作。

## 十三、最终评价

与初始版相比，当前系统在功能、技术和工程管理三个层面都发生了明显跃迁：

- **功能层面**：从 4 个基础 CRUD 页面扩展为覆盖认证、业务、分析、权限、审计和运维的供应链平台。
- **技术层面**：从 Tkinter + 单连接 + 内嵌 SQL，升级为 PySide6 + Matplotlib + 连接池 + 分层服务 + 事务 + 迁移。
- **工程层面**：从个人可运行的脚本，升级为有配置模板、目录规范、版本历史、验证工具和运维说明的团队项目。

最值得概括的一句话是：**初始版解决了“把供应链数据展示和修改出来”的问题，而当前版开始解决“让不同岗位安全、稳定、可追踪地使用这些数据进行业务操作和管理分析”的问题。**
