# 供应链管理系统 Changelog

> 对比基准：旧版 `scm_gui.py` 单文件 Tkinter 版本  
> 当前版本：PySide6 + MySQL 分层桌面系统  

## 1. 总体变化

当前版本已经不再是一个把 UI、SQL、数据库连接、业务逻辑全部写在 `scm_gui.py` 里的单文件程序，而是重构为一个更接近真实桌面业务系统的分层项目。

旧版主要提供：

- Tkinter 主窗口。
- 左侧简单导航。
- 订单管理、库存查询、供应商分析、客户管理四个基础页面。
- 直接在 UI 回调函数中执行 SQL。
- 使用单个数据库连接和游标。
- 少量增删改查能力，无登录、权限、审计、仪表盘和数据库初始化体系。

新版主要新增：

- PySide6 工业控制台风格 GUI。
- 登录认证、角色权限、部门权限和后端权限校验。
- 仪表盘、订单、库存、客户、供应商、用户权限、审计日志等完整模块。
- `core / dao / service / ui / sql / tools` 分层架构。
- MySQL 连接池、事务辅助、初始化 SQL 脚本、兼容旧表结构迁移。
- 审计日志、订单状态审计、用户权限管理。
- Matplotlib 图表、供应商分析可视化、加载动画和性能优化。
- 配置模板、环境变量覆盖、冒烟检查脚本和 README 文档。

## 2. 架构层面的新增

### 2.1 从单文件改为分层架构

旧版 `scm_gui.py` 中，界面代码、SQL、数据库连接和业务规则混在同一个类里。例如按钮事件里直接拼出 SQL、执行查询、提交事务、弹窗提示。

新版拆分为：

| 目录/文件 | 职责 |
| --- | --- |
| `main.py` | 程序入口，初始化 QApplication、数据库对象、登录窗口和主窗口。 |
| `core/config.py` | 读取 `config.ini` 和环境变量，管理数据库与应用配置。 |
| `core/db.py` | MySQL 连接池、游标上下文、事务上下文、数据库对象初始化。 |
| `dao/` | 数据访问层，只负责 SQL 和数据库读写。 |
| `service/` | 业务服务层，负责权限校验、业务规则、事务编排和审计调用。 |
| `ui/` | PySide6 主窗口、样式、通用表格模型和图表组件。 |
| `ui/pages/` | 各业务页面。 |
| `sql/` | 初始化表、视图、存储过程、触发器和演示账号脚本。 |
| `tools/` | 本地冒烟检查脚本。 |
| `docs/` | 项目规范和实施计划文档。 |

这带来的直接好处是：

- UI 层不再直接写 SQL。
- 写操作统一经过 service 层权限校验。
- SQL 更集中，便于审查和优化。
- 事务和审计更容易保持一致。
- 后续新增模块不会继续堆到一个超大 Python 文件里。

### 2.2 legacy 文件降级为参考

旧版 `scm_gui.py` 和 `scm.txt` 仍保留在仓库中，但 README 中明确标注为 legacy 参考，不作为主项目继续扩展。

新版主入口是：

```powershell
python main.py
```

## 3. 技术栈变化

| 项目 | 旧版 | 新版 |
| --- | --- | --- |
| GUI 框架 | Tkinter / ttk | PySide6 |

| UI 风格 | 简单 Tkinter 控件 | 工业调度控制台风格、自定义无边框窗口 |

## 4. 配置与启动能力新增

### 4.1 配置文件模板

新增 `config.ini.example`，用于团队成员复制成本地配置：

```powershell
Copy-Item config.ini.example config.ini
```

`config.ini` 已加入 `.gitignore`，避免真实数据库密码进入仓库。

### 4.2 环境变量覆盖

新版支持通过环境变量覆盖数据库配置：

- `SCM_DB_HOST`
- `SCM_DB_PORT`
- `SCM_DB_USER`
- `SCM_DB_PASSWORD`
- `SCM_DB_NAME`
- `SCM_DB_POOL_NAME`
- `SCM_DB_POOL_SIZE`
- `SCM_ACCOUNT_SET`

这意味着不同成员只要有同结构 MySQL 数据库，数据库名不必固定为 `exp1` 或 `experiment2026`。他们可以在自己的 `config.ini` 中改 `database`，也可以通过 `SCM_DB_NAME` 指向自己的库名。

### 4.3 账套显示跟随数据库名

新版中顶部显示的账套名默认会跟随数据库名。也就是说：

- 如果 `database = my_scm_db`，界面顶部默认显示 `my_scm_db`。
- 如果设置了 `SCM_DB_NAME=my_scm_db`，连接库名和默认账套显示都会跟随它。
- 如果想让显示名和真实库名不同，可以单独设置 `SCM_ACCOUNT_SET`。

## 5. 数据库初始化能力新增

旧版要求数据库中已有全部业务表，程序本身不负责创建应用辅助对象。

新版启动时会尝试初始化应用辅助对象，脚本位于 `sql/`：

| 脚本 | 内容 |
| --- | --- |
| `01_app_tables.sql` | 用户权限表、操作日志表、订单状态审计表。 |
| `02_views.sql` | 库存、销售、供应商、客户相关视图。 |
| `03_procedures.sql` | 仪表盘 KPI 存储过程。 |
| `04_triggers.sql` | 用户变更日志触发器、订单状态审计触发器。 |
| `05_seed.sql` | 演示账号和默认权限数据。 |

此外，`core/db.py` 还加入了旧版表结构兼容逻辑，例如：

- 兼容旧版 `scm_users.password` 明文字段，迁移到 `password_hash`。
- 兼容旧版 `scm_logs.log_time`，补充新版 `created_at` 字段。

## 6. 登录与权限系统新增

旧版没有登录，也没有权限控制。任何能打开程序的人都可以执行所有按钮对应操作。

新版新增完整登录流程：

- `ui/login_dialog.py`：登录界面。
- `service/auth_service.py`：认证和会话对象。
- `dao/auth_dao.py`：用户查询、权限更新、审计写入。
- `sql/05_seed.sql`：默认演示账号。

默认演示用户包括：

| 用户 | 角色 | 部门 | 权限说明 |
| --- | --- | --- | --- |
| `admin` | admin | 总管理 | 拥有全部权限。 |
| `David` | manager | 采购部 | 本部门完全控制，其他业务只读。 |
| `Tom` | manager | 销售部 | 本部门完全控制，其他业务只读。 |
| `Jerry` | manager | 客户管理部 | 本部门完全控制，其他业务只读。 |
| `Marry` | staff | 采购部 | 本部门查看和插入。 |
| `Jack` | staff | 销售部 | 本部门查看和插入。 |
| `Mike` | staff | 客户管理部 | 本部门查看和插入。 |

新增的权限规则包括：

- admin 后端强制拥有全部权限，不依赖数据库权限位。
- manager 可查看其他部门业务，但只能完全控制本部门模块。
- staff 只能查看和插入本部门数据，不能更新和删除。
- 用户权限管理只允许 admin 执行。
- UI 按钮隐藏或禁用只是体验层，后端 service 仍会再次校验权限。

## 7. 审计日志新增

旧版没有操作审计。

新版新增：

- `scm_logs`：记录登录、用户权限变更、订单创建、订单状态更新、库存补货、客户增删改等关键操作。
- `scm_order_audit`：通过触发器记录订单状态变化。
- `ui/pages/logs_page.py`：审计日志查询页面。
- `service/admin_service.py`：审计日志查询权限控制。

重要改进：

- 关键写操作和审计日志写入被放入同一个事务中。
- 业务写入成功但日志失败时会整体回滚，避免数据和审计不一致。

## 8. UI 与交互新增

### 8.1 PySide6 工业控制台风格

新版从 Tkinter 默认控件升级为 PySide6，并新增统一 QSS 样式：

- 深色左侧导航栏。
- 暖白顶部工具栏。
- 暖灰内容区。
- 铜色主操作按钮。
- 数据密集型表格布局。
- 自定义 KPI 卡片和分析卡片。

相关文件：

- `ui/main_window.py`
- `ui/styles.py`
- `ui/table_model.py`
- `ui/analytics_widgets.py`

### 8.2 自定义无边框窗口

新版去掉系统默认标题栏，把最小化、最大化、关闭按钮嵌入应用 UI。

新增能力：

- 顶部栏拖动窗口。
- 双击顶部栏切换最大化。
- 自定义关闭、最大化、最小化按钮。
- 轻微圆角外观。
- 无边框窗口边缘和角落可拖动缩放。
- 修复缩小时闪烁问题。

### 8.3 懒加载页面

主窗口不再一次性创建所有页面，而是切换到页面时才加载对应内容。

这减少了首次进入系统后的阻塞时间，也让页面初始化更加可控。

## 9. 仪表盘模块新增

旧版没有仪表盘。

新版新增 `DashboardPage`：

- 总订单数。
- 累计销售额。
- 客户数量。
- 缺货预警数量。
- 销售趋势数据。
- 低库存快照表。

数据来源包括：

- `sp_scm_dashboard_kpis` 存储过程。

如果存储过程不可用，service 层会尝试 fallback 普通 SQL 查询。

## 10. 订单管理模块增强

旧版订单模块提供：

- 订单列表。
- 按订单号搜索。
- 查看订单明细。
- 更新订单状态。

新版增强为：

- 订单列表通过 DAO/service 分层查询。
- 查询支持订单号或客户名称。
- 订单明细显示零件、供应商、数量、金额、行状态等更完整字段。
- 新增模拟下单能力。
- 下单时会校验库存并扣减库存。
- 下单、库存扣减和审计日志在同一事务内完成。
- 状态更新只允许 `O`、`F`、`P`。
- 状态更新会锁定订单行，避免并发状态异常。
- 重复设置相同状态不会写入无意义审计。
- 订单状态变化由触发器写入 `scm_order_audit`。

## 11. 库存调度模块增强

旧版库存模块主要是库存查询。

新版增强为：

- 低库存筛选。
- 库存状态分类：`low`、`warning`、`normal`。
- 补货操作。
- 补货权限校验。
- 补货数量、零件 ID、供应商 ID 后端校验。
- 零件-供应商关系不存在时明确提示。
- 补货和审计日志同事务提交。
- ABC 库存分类展示。

## 12. 客户管理模块增强

旧版客户模块提供：

- 客户列表。
- 客户搜索。
- 添加客户。
- 删除客户。

新版增强为：

- 客户列表展示客户 ID、客户名称、国家、电话、余额、订单数、累计收入。
- 支持客户 ID 或名称查询。
- 支持新增、编辑、删除客户。
- 后端校验客户名称、地址、电话、国家 ID 等字段。
- 客户新增、更新、删除都经过 service 权限校验。
- 客户变更和审计日志同事务提交。
- 客户列表查询经过优化：先限制首屏客户，再聚合这些客户的订单数据，避免全量视图聚合导致加载慢。
- 表格渲染优化：禁用编辑、整行选择、关闭换行、固定行高、刷新状态提示。

## 13. 供应商分析模块增强

旧版供应商模块只是一个统计表格：供应商 ID、名称、国家、供应零件数、平均成本。

新版增强为完整分析页面：

- 供应商 KPI：活跃供应商、优质供应商、供应零件总数、月均采购额。
- 供应商评分与排行。
- 各供应商供应量柱状图。
- 供应商国家分布环形图。
- 采购额趋势折线图。
- 图表 hover 提示。
- 图表动画。
- 中文字体兼容处理。
- 后台线程加载供应商分析数据，避免点击模块时 UI 主线程卡住。
- 顶部不定进度条和加载状态提示。
- DAO 查询优化，减少首屏全量聚合。

## 14. 用户权限模块新增

旧版没有用户管理。

新版新增 `UsersPage`：

- 查看系统用户。
- 修改用户权限。
- 只有 admin 可以进入和操作。
- staff 的 update/delete 权限会被后端强制关闭。
- admin 权限位会被后端强制保持全开，避免管理员被降权导致系统不可管理。
- 权限变更写入审计日志。

## 15. 审计日志模块新增

新版新增 `LogsPage`：

- 展示日志 ID、用户、模块、动作、详情、时间。
- manager 和 admin 可以查看审计日志。
- staff 无权查看。
- 日志按时间倒序展示。

## 16. 数据访问与事务能力增强

### 16.1 参数化查询

新版 DAO 中的用户输入均通过参数化查询传入，例如：

```python
db_cursor.execute(sql, params)
```

这比旧版把 SQL 写在 UI 回调里更容易统一检查，也降低 SQL 注入风险。

### 16.2 连接池

旧版在应用启动时创建一个连接和一个游标。

新版通过 `mysql.connector.pooling.MySQLConnectionPool` 创建连接池，每次查询使用上下文管理器获取连接并自动关闭，降低连接泄漏风险。

### 16.3 事务上下文

新版新增 `transaction()` 上下文管理器：

- 成功时 commit。
- 异常时 rollback。
- 最终关闭游标和连接。

订单创建、库存补货、客户变更、用户权限变更等写操作均已使用事务。

## 17. 性能优化新增

当前版本已经针对几个明显慢点做了优化：

- 客户管理：避免从客户视图全量聚合后再 `LIMIT`，改为先限制客户，再聚合订单。
- 供应商分析：后台线程加载，避免 UI 卡死。
- 供应商分析：首屏查询限制数据范围，减少全量聚合。
- 页面懒加载：未点击的页面不提前创建。
- 表格渲染：部分页面关闭编辑、固定行高、减少重复布局。
- 图表中文字体配置缓存，提升图表稳定性。

## 18. 本地检查与质量保障新增

新增 `tools/smoke_check.py`，用于轻量检查：

- Python 文件是否能通过 AST 语法解析。
- 仓库中是否残留旧数据库密码或旧路径片段。
- UI 层是否存在直接 SQL 字符串。

README 中也补充了本地检查命令：

```powershell
python -m compileall -q -x "(\.venv|\.git|\.idea|__pycache__)" .
python tools\smoke_check.py
```

## 19. 安全性提升

相较旧版，新版主要安全提升包括：

- 不提交真实 `config.ini`。
- 数据库密码不硬编码在主项目代码中。
- 登录密码使用 SHA2 摘要匹配，不再依赖明文密码字段。
- 写操作必须通过 service 权限校验。
- admin、manager、staff 权限边界明确。
- 关键操作写入审计日志。
- UI 禁用按钮只是体验层，后端仍会拒绝越权调用。
- 冒烟检查会扫描旧数据库密码片段。

## 20. 仍然需要注意的点

当前版本相比旧版已经完整很多，但仍有一些工程注意事项：

- 程序启动时会自动创建视图、存储过程和触发器，因此 MySQL 用户需要相应权限。
- 课程核心表如 `Customer`、`Orders`、`Lineitem`、`Part`、`PartSupp`、`Supplier`、`Nation` 仍需提前导入。
- `Orders.Orderkey`、`Customer.Custkey` 当前仍使用应用逻辑生成，后续如要支持高并发，最好改为更稳的自增或序列表方案。
- 部分数据库查询已经做了首屏优化，但大数据量场景下还可以继续补索引和分页。
- `scm_gui.py` 仍保留 legacy 参考，但不应继续作为主项目扩展。

## 21. 简要结论

新版相较 `scm_gui.py` 单文件版本，不只是换了一个界面框架，而是完成了从课程演示式 Tkinter CRUD 工具到分层 PySide6 桌面业务系统的升级。

最核心的新增价值是：

- 架构可维护。
- 权限可控制。
- 操作可审计。
- 数据库初始化可复现。
- 界面更接近真实业务系统。
- 慢查询和页面卡顿开始得到针对性优化。
- 团队成员可通过配置文件或环境变量使用自己的数据库名启动系统。
